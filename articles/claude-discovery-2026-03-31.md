# Claude Code 今日の発見 — 2026-03-31

「派手なデモより、地味に毎日動く仕組み」——今日見つけた事例はどれも、一発芸ではなく実務に定着した自動化の話が中心だった。

## 15エージェントのパイプラインで「2週間の仕事を4時間」に

あるGTM（Go-to-Market）支援のコンサルタントチームが、Claude Codeで15種類の専門エージェントを連携させたパイプラインを構築した。YouTube動画の時系列分析、Redditからの顧客インサイト抽出、X（旧Twitter）の高エンゲージメント投稿分類——これらを「リサーチフェーズ」として並列実行させ、その後にポジショニング、メッセージング、コピーライティングの各エージェントが順次処理する。各段階には「品質管理マネージャー」役のエージェントが挟まり、新規性とインパクトの2軸で出力をスコアリングする。

結果として、かつて2週間かかっていた案件が4時間で完成するようになり、このチームは料金を4倍に引き上げた。3,000社以上の小規模エージェンシーがこの手法を導入しているという。顧客の実際の言葉をそのまま活用するため、テンプレ出力と比べて3倍のエンゲージメント率を達成しているとのこと。YOMIKO的には、競合分析やターゲットインサイト抽出のフェーズで同様のリサーチエージェント群を組むと、提案資料の初稿作成が激変しそう。

## 前提知識ゼロで大規模プロジェクトに途中参画、バグ修正を完遂

TVer（ティーバー）は日本のテレビ番組を無料配信する動画サービスで、iOSアプリを含む複数のプラットフォームを運営している。同社のiOSエンジニア福島氏は、プロジェクト固有の知識がほぼない状態で途中参画し、Claude Codeのスキル機能を使って「分析→実装→PR作成」の3ステップを標準化した。

QAから報告された不具合を調査し、原因特定と修正方針をまとめたIssueを自動生成。そのIssueをもとに実装を行い、最後にAndroid実装との比較表を含む詳細なPRを生成する——この流れをスキルとして定義した。バグ修正は明確なゴールがあり、スコープが限定的なためAI活用と相性が良いという。途中参画メンバーでも一定品質のアウトプットが可能になり、プロジェクトのリリースを成功させた。

## Polymarketの自動取引システムを「会話だけで」構築

Polymarketは暗号資産を使った予測市場プラットフォームで、政治や経済イベントの結果にベットできるサービスだ。あるエンジニアがClaude Codeとの単一の会話セッション内で、完全な自動取引システムを構築した。価格監視（5分ごとにポートフォリオのP&Lをトラッキング）、アービトラージ検出（複数候補の市場でYES価格の合計が1ドル未満なら保証された利益機会）、取引実行（1取引最大20ドル、1日最大50ドルの安全装置付き）——すべてが会話ベースで実装された。

初週の成績は8件の決済済み取引で75%の勝率、9.6%の実現ROI。NegRiskアービトラージと呼ばれる戦略では36.2%の最高ROIを達成した。「退屈な戦略ほど安定している」という結論も興味深い。Claude Codeで金融システムを組む事例としては最も具体的なものの一つ。

## 広告分析が「3〜5時間」から「体感7分」に

Rimo（リモ）は音声文字起こしやAI議事録サービスを提供する日本のスタートアップだ。同社のマーケティング責任者・佐伯氏は、Google広告の分析業務をClaude Codeで自動化した。従来は管理画面で数値を手動確認し、キーワードや検索語句を精査し、改善案を人間が考察——これに最低3時間、悪い時は4〜5時間かかっていた。

新しいフローでは、BigQueryに格納された広告データをClaudeが自動取得し、状況の要約と考察を生成、改善提案まで作成する。人間がやるのは提案の採否判断だけ。Google広告APIで全自動化も技術的には可能だが、「暴れたときのリスク」を考慮して、現段階では人間チェックを挟む二段階方式を選択している。この「提案はAI、実行判断は人間」という切り分けは、広告運用以外でも汎用性が高い。

## Slack × GitHub × Claudeでチャットボット改善サイクルを自動化

社内チャットボットの改善は、ユーザーからの「このBot使えない」という声を拾い、原因を調査し、修正する——という地道な作業の繰り返しだ。ある開発チームは、SlackとGitHub、Claude Code Actionsを連携させて、このサイクルをほぼ自動化した。

ユーザーがSlackでBotにメンション→Botが回答→ユーザーがGood/Badボタンで評価→フィードバックがGitHub Issueに自動変換→Claude Code Actionsが分析・改善提案を実施→開発者がレビュー後に修正。回答フッターには「version: main-abc1234」のようなバージョン情報が記録され、問題の再現が容易になっている。フィードバックはhallucinationかretrieval_failureかといったカテゴリ分類付きで収集され、傾向分析と優先度判断が可能になる。

人間が手を動かすのは「フィードバック入力」と「最終レビュー」だけ。Issue生成からトリアージまでが自動化されているため、開発者は調査不要で実装に集中できる。

## 「毎朝7時に勝手に動く」——CPOが作った11個の日常自動化

アソビュー（体験予約のマーケットプレイスを運営する日本企業）のCPO・横峯氏が、Claude Codeで構築した11個の自動化ジョブについて書いている。毎朝7時になると勝手に動き出し、Slackを開くと結果が待っている——という仕組みだ。

「デモ映えする一発芸と、業務に定着する仕組みは別物」という指摘が印象的で、派手なアプリ開発デモではなく、地味だが毎日使い続ける実用的な自動化にフォーカスしている。具体的な11個の内容は記事本文で詳述されているが、非エンジニアのCPOが実務レベルで自動化を回している点が重要だ。

## Claude Code全社導入、決断までに1年かかった反省

Gemcook（ジェムクック）はWeb・アプリ開発を手がける日本の制作会社だ。同社は2026年2月にClaude Codeを全社導入したが、その意思決定プロセスを公開している。社長から「AIリテラシーを標準化しないと取り残される」という危機感が示され、Gemini Code Assist、Cursor、Windsurf、Kiro、Devinなど複数の候補を比較検討した結果、実装品質と速度のバランス、開発界隈での普及度、Teamプランの運用しやすさを理由にClaude Codeを選定した。

導入後、数日で数万行のコードを生成したメンバーが出現するなど活用は進んでいるが、著者は「Coding Agent群雄割拠時代での判断遅れ」を反省している。検証開始から導入決定まで約1年かかったが、1〜2ヶ月での意思決定が望ましかったという。「明日もっといいツールが出るかもしれないけど、今は今の最適解を選ぶ」——この判断基準は、ツール選定に迷う組織への示唆になる。
